# caysonverse 기획문서 (설계 스펙)

- 작성일: 2026-07-14
- 상태: 1차 버전(v1) 확정 — 구현 착수 기준 문서
- 작성 경위: 발주자(AI 강사)와의 브레인스토밍 대화로 요구사항을 확정하고, 기술 검증 리서치(라이브러리 버전·네트워크 설계·에셋 라이선스)를 거쳐 작성함

## 1. 프로젝트 개요

**caysonverse**는 웹 브라우저에서 링크 클릭만으로 접속하는 3D 메타버스다. 접속자는 닉네임과 캐릭터를 고르고 3D 공간에 입장해, 다른 접속자들과 함께 돌아다니고 머리 위 말풍선으로 대화한다.

- **운영자**: AI 강사 (1인 운영)
- **용도**: ① 강의/수업 공간 ② 수강생·커뮤니티 상시 라운지
- **핵심 가치**: 설치·가입 없는 즉시 입장, 실시간 현장감(아바타+말풍선), 강의 도구(공지·강퇴, 향후 화면공유)

### 1.1 v1 범위 (이번에 만드는 것)

| 기능 | 내용 |
|---|---|
| 게스트 입장 | 닉네임(2~12자) + 프리셋 캐릭터 4종 + 색상 선택 → 즉시 입장. 회원가입/DB 없음 |
| 3D 월드 | 하나의 맵에 라운지 + 강의실 두 구역, 걸어서 이동 |
| 실시간 동기화 | 최대 100명 동접, 아바타 위치·방향 실시간 표시 |
| 말풍선 채팅 | 채팅 시 머리 위 말풍선 6초 + 접이식 채팅 로그 |
| 이모지 리액션 | 👍 ❤️ 😂 👏 🎉 🙋 6종, 머리 위 3초 표시 |
| 강사용 관리 | 관리자 코드 입력 → 전체 공지 배너, 사용자 강퇴 |
| 모바일 | 가상 조이스틱 + 터치 카메라, 저사양 자동 최적화 |
| 재접속 | 순단 시 20초 내 자동 복귀 (같은 위치·캐릭터) |

### 1.2 v2 이후로 미루는 것

- 화면 공유/발표 (WebRTC) — 강의실 스크린 오브젝트는 v1에 미리 배치
- 음성 대화
- 계정/영속 저장(DB), 지속 밴
- 룸 분할·수평 확장 (100명 초과)

## 2. 사용자 시나리오

### 2.1 수강생 (게스트)
1. 강사가 공유한 링크 클릭 → 입장 화면
2. 닉네임 입력, 캐릭터 4종 중 선택, 색상(파스텔 팔레트) 선택 → [입장하기]
3. 라운지 스폰 지점에 등장. PC는 WASD/방향키 + 마우스 드래그 카메라, 모바일은 좌하단 조이스틱 + 터치 드래그
4. Enter(또는 채팅 버튼)로 채팅 → 내 머리 위 말풍선 + 로그 기록
5. 이모지 버튼으로 리액션
6. 강의 시간엔 강의실 구역으로 걸어가 착석 지점 근처에서 수업 참여

### 2.2 강사 (관리자)
1. 입장 화면에서 닉네임 등 입력 + [관리자 코드] 필드에 코드 입력
2. 서버가 환경변수 `ADMIN_CODE`와 대조(클라이언트 번들에 코드 미포함, 시도 5회/분 제한)
3. 관리자 UI 활성화: 전체 공지 작성(모든 화면 상단 배너, 늦게 입장해도 보임), 접속자 목록에서 강퇴
4. 강퇴된 사용자는 세션 denySet에 등록되어 즉시 재입장 차단(서버 재시작 전까지)

```
브라우저 (React 19 + R3F 9)                Railway 단일 서비스 (Node 24)
┌──────────────────────────────┐  WebSocket  ┌──────────────────────────────┐
│ 입장화면(닉네임·캐릭터·색·관리자코드)   │◄──────────►│ Colyseus WorldRoom (단일 룸)      │
│ 3D 월드 (R3F, 60fps 렌더)         │ @colyseus/sdk│  · 서버 권위 상태, 10Hz patch      │
│ UI 오버레이(채팅·이모지·조이스틱·배너) │             │  · 이동 검증·채팅·공지·강퇴          │
│ 네트워크 상태는 React 밖 mutable Map│             │ express: 클라 dist 정적 서빙+SPA    │
└──────────────────────────────┘             └──────────────────────────────┘
                                               DB 없음(메모리만), PORT=env, 헬스체크
```

**모노레포 (npm workspaces)**

```
caysonverse/
├─ docs/           # 이 기획문서 등
├─ shared/         # 스키마 클래스(서버 전용 실행) + 메시지 타입/상수 (브라우저 안전 엔트리 분리)
├─ server/         # Colyseus 서버 (CJS/NodeNext, 데코레이터 tsconfig)
├─ client/         # Vite 8 + React 19 + R3F
└─ loadtest/       # @colyseus/loadtest 봇
```

- shared의 엔트리 분리 원칙: `shared/schema`(데코레이터 실행 — 서버 전용, 클라는 `import type`만), `shared/messages`·`shared/constants`(플레인 TS — 클라·서버 공용). 클라이언트 번들에 스키마 런타임 코드가 들어가지 않도록 배럴(barrel) 재수출 금지.
- 개발: vite dev(5173)에서 `/matchmake`·WS를 localhost:2567로 프록시, 서버는 tsx watch.
- 프로덕션: `vite build` → 서버가 `client/dist` 정적 서빙(+SPA fallback, `/matchmake`·모니터 경로 제외).

## 5. 네트워크 설계 (수치 검증됨)

- **patchRate = 100ms (10Hz)** — 기본 20Hz의 절반 비용. 보간으로 체감 차이 없음. 부족하면 66ms로 한 줄 변경
- **입력 모델**: 클라이언트가 자기 위치를 로컬에서 통합(조이스틱 즉각 반응) → 이동 중 10Hz로 `move {x, z, yaw}` 전송 + 정지 시 마지막 1회. 서버 검증:
  - NaN/비유한값 거부
  - 변위 ≤ maxSpeed × 경과시간 × 1.5 (순간이동 차단)
  - 보행 가능 AABB로 클램프 (맵 밖 차단)
  - 30 msg/s 초과 드롭
  - y는 서버 고정 (비행 원천 차단)
- **Player 스키마**: `nickname`(string, 입장 시 1회), `character`(uint8), `tint`(uint8), `x`·`z`(float32), `yaw`(float32), `connected`(boolean). 애니메이션 상태 필드 없음 — 클라가 보간 속도로 walk/idle 유도. 채팅은 상태 저장 금지(안티패턴)
- **대역폭**: 전원(100명) 이동 시 클라당 다운 ~24KB/s(≈200kbps) — 모바일 OK. 강의 모드(대부분 정지)는 델타 인코딩 덕에 5~20% 수준. 초과 시 int16 양자화 예비안
- **원격 보간**: 스냅샷 링버퍼(~10개) + 150ms 렌더 지연, yaw 최단호 lerp, 버퍼 고갈 시 250ms 외삽 후 idle, 3m 초과 이탈 시 스냅
- **기능별 전송 메커니즘**:

| 기능 | 메커니즘 | 제한 |
|---|---|---|
| 채팅 | `broadcast` | 200자, 5초당 3회(초과 시 조용히 드롭+개인 알림) |
| 이모지 | `broadcast` (uint8 인덱스) | 500ms당 1회 |
| 공지 | 루트 상태 필드 (`announcement`, `announcedAt`) | 늦게 입장해도 보여야 하므로 상태 |
| 강퇴 | `client.leave(4001)` + 메모리 denySet | onJoin에서 denySet 확인 |

- **재접속**: `onDrop`에서 `allowReconnection(client, 20)` — 20초 창(유령의 정원 점유 최소화). `reconnectionToken`은 sessionStorage, 실패 시 localStorage의 닉네임·캐릭터로 조용히 재입장. 접속 끊긴 아바타는 50% 투명 렌더
- **정원**: maxClients 110 (100 + 재접속 여유), 초과 시 "정원이 가득 찼습니다" 안내

## 6. 렌더링·성능 설계 (100 아바타, 모바일 30fps 목표)

- 캐릭터 GLB 4종을 1회 로드 → 플레이어마다 **SkeletonUtils.clone()** (일반 clone은 SkinnedMesh 파손)
- **색상 틴트**: 클론별 `material.clone()` + `material.color.set(tint)` — 텍스처는 공유(GPU 메모리 불변). multiply 틴트는 어두워지기만 하므로 프리셋 팔레트는 **밝은 파스텔 8색**(순백 = 원본)
- **AnimationMixer 스로틀**: 카메라 <10m 매 프레임 / 10~25m 3프레임당 1회(누적 delta 전달) / >25m·프러스텀 밖·정지·착석 시 정지. 강의실 최악 케이스(전원 착석·전원 화면 안)를 이 규칙이 구제
- **닉네임표·말풍선**: canvas 텍스처 THREE.Sprite (drei `<Html>` 100개는 모바일 사망). 20m 밖 닉네임 숨김, 동시 말풍선 30개 캡
- `<Canvas dpr={[1, 1.5]}>`, 모바일 그림자 전면 off + 블롭 섀도 쿼드, 데스크톱은 1024px directional 1개까지
- 조명: hemisphere + directional 2개, Lambert/Toon 계열 머티리얼 (멀티라이트 PBR 금지)
- SkinnedMesh `boundingSphere` 수동 지정 (애니메이션 중 팝아웃 방지)
- **네트워크 상태를 React state에 넣지 않는다** — React 밖 mutable Map에 기록, useFrame에서 ref로 읽어 Object3D에 직접 반영. React state는 입장/퇴장(마운트/언마운트)과 UI 전용. (100명×10Hz = 초당 1,000 setState 방지 — 최다 실수 포인트)

## 8. 보안·어뷰징 대응 (익명 게스트 공간)

| 위협 | 대응 |
|---|---|
| 순간이동/스피드핵 | 서버 변위 클램프 + AABB + NaN 거부 |
| 채팅 도배 | 5초당 3회 + 200자 + 제어문자 제거 |
| 부적절 닉네임 | 길이 2~12자 + 문자셋 필터 (onJoin) |
| 관리자 코드 무차별 대입 | 서버 측 비교만(타이밍 세이프) + 시도 5회/분 제한 (IP별) |
| 강퇴 후 재입장 | 세션 denySet — 완전 차단은 계정 필요(v2 한계로 수용) |

**알려진 한계 (v1 수용):**
- 클라이언트 IP는 프록시 헤더(x-forwarded-for 등)로만 얻을 수 있어, **프록시 없는 환경(로컬 등)에서는 시도 제한이 전역 카운터로 동작**한다. 이 경우 공격자가 분당 5회 오답을 보내면 진짜 강사도 로그인이 잠길 수 있다. **Railway는 프록시 뒤에서 동작하므로 실배포에서는 IP별 제한이 정상 작동**한다 — 배포 문서에 명시할 것. **단, x-forwarded-for는 클라가 좌측 항목을 위조할 수 있으므로 서버는 신뢰 프록시(Railway 단일 홉)가 append한 _맨 오른쪽_ 홉을 실제 클라 IP로 선택한다**(server `clientIp.ts`). 신뢰 프록시가 2개 이상으로 늘면 이 "마지막 홉 = 클라" 가정을 재검토해야 한다.
- 강퇴 denySet은 **닉네임 키만** 사용한다(IP는 밴 키로 쓰지 않음). Railway 뒤에서 IP는 교실 전체가 공유하는 NAT 공인 IP이므로, IP로 밴하면 한 명 강퇴에 같은 IP를 쓰는 반 전체가 차단된다(F7). 닉네임 키는 강퇴당한 사용자의 동일 닉네임 재입장을 막으면서 다른 학생을 과차단하지 않는다. 대신 강퇴당한 사용자가 닉네임을 바꾸면 재입장이 가능하다(계정 없는 v1의 구조적 한계). IP는 관리자 코드 시도 제한(brute-force limiter)에만 계속 쓰인다.

## 9. 에러 처리·안정성

- 연결 끊김: "재연결 중..." 토스트 → reconnectionToken으로 자동 복귀 → 실패 시 조용히 신규 재입장 (지수 백오프)
- 서버 배포/크래시: 전원 접속 종료 + 상태 소실 (DB 없음 — v1 수용 리스크). 강의 시간 외 배포 원칙
- 정원 초과: 입장 화면에서 안내
- 헬스체크: express 라우트 (WS 엔드포인트 아님)

## 14. 의자 앉기/일어서기

### 14.1 상호작용
- **PC**: 강의실 의자 1.5m 이내 접근 시 "**E** 앉기" 안내 → E로 착석. 앉은 상태에서 E 또는 이동키 → 일어섬
- **모바일**: 근접 시 [앉기] 버튼 표시 → 착석 후 [일어서기] 버튼. 조이스틱을 밀어도 일어섬
- 앉으면 의자 위치에서 **스크린 방향을 향해 착석**, 일어서면 의자 뒤 통로 쪽(각 좌석별 사전 정의된 하차 지점)에 섬
- 앉은 채로 채팅·이모지·발표 시청 전부 가능

### 14.2 애니메이션 (검증 완료)
KayKit GLB 내장 클립 사용: `Sit_Chair_Down`(착석 동작) → `Sit_Chair_Idle`(앉은 자세 유지) / `Sit_Chair_StandUp`(기립 동작). 4종 캐릭터 공통 스켈레톤이므로 클립 세트 하나로 전부 동작.

### 14.3 서버 권위 (v1 원칙 유지)
- 좌석 데이터는 shared/worldMap의 강의실 의자 배치에서 파생: **13석** (학생 12 + 강사석 1), 좌석별 {위치, 착석 방향, 하차 지점}
- Player 스키마에 `seatIndex`(int8, -1=미착석) 추가 — 착석 상태가 전원에게 동기화 (100명 착석 장면 가능)
- 착석 검증: 좌석 인덱스 범위, **1의자 1인**(선착순, 탈락자에게 "이미 사용 중" 개인 알림), 의자와의 거리 ≤ 1.5m
- 착석 중 이동 메시지는 무시(silent drop). 일어서기는 명시적 Stand 메시지
- 좌석 반환: 일어서기·영구 퇴장·강퇴 시. 순단 재접속(20초 창)은 좌석 유지
- 하차 지점은 장애물과 겹치지 않음을 맵 불변 테스트로 보증

## 15. 화면 공유/발표

### 15.1 아키텍처
- **LiveKit Cloud**(관리형 SFU) 사용: 강사 1명이 화면+마이크 publish → 최대 100명 subscribe (지연 <0.3초)
- Colyseus(이동·채팅)와 LiveKit(미디어)는 독립 — 한쪽 장애가 다른 쪽에 전파되지 않음
- **토큰은 caysonverse 서버가 발급**: LiveKit API 키/시크릿은 서버 env에만 존재. 관리자에게만 publish 권한 토큰, 일반 사용자는 subscribe 전용 (v1 관리자 검증과 동일한 서버 권위 원칙)
- 발표 중 여부는 Colyseus 루트 스키마 상태로 동기화 (공지 배너 패턴 — 늦게 입장해도 즉시 표시)
- 서버 env 3개 추가: `LIVEKIT_URL`, `LIVEKIT_API_KEY`, `LIVEKIT_API_SECRET` (Railway Variables)

### 15.2 발표자(강사) UX
- 관리자 패널 [발표 시작] → 브라우저 화면 선택 창 → **화면+마이크** 동시 송출, 버튼은 [발표 종료]로 전환 + "발표 중 🔴" 표시
- 종료: 버튼 클릭 / 브라우저 "공유 중지" / 강사 접속 종료·새로고침 — 어느 경로든 전원 스크린이 원상 복귀 (LiveKit 이벤트 + Colyseus 상태 이중 정리)
- 동시 발표는 1명 (서버가 이중 발표 거부)

### 15.3 시청자 UX
- 발표 시작 시: 강의실 3D 스크린에 실시간 비디오 텍스처 + 채팅 로그에 "📽️ 발표가 시작되었습니다" 시스템 메시지
- **자동재생 정책 대응**: 소리 포함 재생은 사용자 제스처가 필요할 수 있음 → 필요 시 "발표 시청하기 ▶" 배너 1탭으로 소리까지 재생
- 스크린 클릭/탭 → 전체화면 오버레이 (X·ESC로 닫기). 시청 중 이동·채팅·이모지 정상 동작
- 수강생 재접속 시 발표 중이면 자동 재연결

### 15.4 엣지 케이스
| 상황 | 처리 |
|---|---|
| 강사 재접속 | 발표 종료 처리 → 재시작 필요 (상태 잔재 없이 정리) |
| LiveKit 연결 실패 | 스크린 평소 상태 + "발표 연결에 실패했습니다" 안내, 다른 기능 정상 |
| 이중 발표 시도 | 서버 거부 |
| 100명 시청 부하 | SFU 부담 — 클라는 수신 1트랙만 |

### 15.5 비용·버전 (2026-07-16 리서치 확정 — 이 수치가 보류 결정의 근거)

- **무료 티어(Build)로는 실강의 불가**: 동시 100 WebRTC 연결 한도(강사+수강생100=101 초과), 월 5,000 참가자-분(2시간 강의 1회 = 101명×120분 = 12,120분으로 초과), 50GB 전송 — **한도 초과 시 과금이 아니라 신규 요청 실패(강의 도중 컷)**. 개발·소규모 테스트 전용.
- **실운영 = Ship 플랜 $50/월**: 150,000 참가자-분 포함(월 4회 강의 48,480분 → 여유), 250GB 포함 + 초과 $0.12/GB. 1080p15fps 화면공유(기본 2.5Mbps) 기준 강의당 90~230GB → **월 4회 강의 시 총 $63~130/월 (강의당 $16~33)**.
- **절감 레버(재개 시 설계 반영)**: 3D 스크린 텍스처는 저화질 시뮬캐스트 레이어 구독(`setVideoDimensions` 720p/360p), 전체화면 오버레이만 1080p → egress 최대 4배 절감, 월 $50~65 수준 가능. 주의: WebGL 텍스처 렌더링은 `adaptiveStream` 자동 최적화가 작동하지 않으므로(off-DOM 엘리먼트 = invisible 판정 → 트랙 일시정지 버그) `adaptiveStream: false` + 수동 레이어 지정 필수.
- 검증된 SDK (재개 시 사용): livekit-client 2.20.1(React 19 무관, 바닐라 TS), livekit-server-sdk 2.17.0(Node 24 CJS 번들 호환, `toJwt()`는 **async**), @livekit/components-react는 불필요(DOM 그리드용). 토큰 TTL 기본 6h, 연결 중 자동 갱신. 대안 비교: Daily.co ~$194/월(더 비쌈), Cloudflare Realtime ~$46/월(저렴하나 저수준 — 솔로 운영자 개발량 과다).

## 21. 모바일 1인칭 D-패드 (2026-07-16 발주자 요청)

- **문제 (진단 확정)**: 모바일 1인칭에서 조이스틱 조종 시 방향이 심하게 왔다 갔다 함. 원인은 시선 동행(20장)과 아날로그 스틱의 상호작용 — 키보드는 8방향 이산 입력이라 "정확한 전진"(회전 0)이 존재하지만, 조이스틱은 엄지가 전방에서 ~2.4°(림에서 약 2.3px)만 어긋나도 시선이 최대 속도(143°/s)로 조향되는 구조. 엄지 떨림(±3~10°)이 초당 수회 부호를 바꾸며 시선이 17~22° 폭으로 왕복 → 보고 증상과 일치. 부수 피해: 요동하는 yaw가 네트워크로 방송되어 타인 화면에서도 아바타가 떨려 보임. (조이스틱 터치가 시선 드래그로 새는 이벤트 버그는 없음 — 검증 완료)
- **해결 (발주자 안 채택)**: 모바일 **1인칭일 때만** 좌하단 조이스틱을 **십자 D-패드**로 전환 — 입력을 키보드와 동일한 8방향 이산값으로 만들어 검증된 데스크톱 거동(▲=회전 0 직진, ◀=곡선 선회)을 그대로 재사용. 3인칭·오버뷰에서는 기존 조이스틱 유지(카메라 자동 회전이 없어 아날로그가 유리)
- **D-패드 사양 (슬라이드 8방향)**:
  - 겉보기는 ▲▼◀▶ 십자 버튼(각 ≥44px), 실동작은 단일 터치 존 — 엄지를 떼지 않고 슬라이드하면 터치 각도를 8섹터(45°)로 양자화해 대각(↖↗↘↙) 포함 8방향 인식. 활성 방향 버튼 하이라이트(대각은 인접 2버튼)
  - **경계 채터링 방지**: 섹터 전환에 각도 히스테리시스 ±10°(업계 표준 진입/해제 문턱), 중심 반경 데드존(기존 force 0.15 규칙 상당) 이내는 정지
  - 산출 intent는 키보드와 동일한 {-1,0,1} 성분으로 기존 합산 경로에 주입 — 시선 동행·이동·네트워크 로직 무변경 (순수 입력단 교체)
- **전환 규칙**: 뷰 모드가 1인칭이면 D-패드, 그 외 조이스틱 (기존 zustand 뷰 플래그로 UI 전환). 시선 미세 조정은 기존처럼 D-패드 밖 화면 드래그

## 23. 채팅 음성 낭독 (TTS) (2026-07-16 발주자 요청)

- **기능**: 채팅 메시지가 말풍선과 함께 **음성으로도 낭독**됨 ("안녕" → 말풍선 + 음성). 모든 채팅 메시지 대상, 단 **내 캐릭터 근처(말풍선 가독 거리 상당)의 발신자만** — 강의실 다인원 소음 방지. 내 메시지는 거리 0이므로 항상 낭독(발주자 시나리오 그대로)
- **기술**: 브라우저 내장 Web Speech API(SpeechSynthesis, ko-KR 음성 우선) — **서버/네트워크 무변경, 추가 비용 0**. 음질은 기기 내장 수준(발주자 고지: 유료 클라우드 TTS는 과금이라 배제). 음성 목록 비동기 로딩(voiceschanged) 대응
- **동시 다발 제어**: 직렬 큐 + 대기 상한(초과분 오래된 것부터 드롭), 낭독 길이 상한(장문은 앞부분만). 거리 비례 볼륨 감쇠는 구현 가능 범위에서(utterance volume) — 구현자 판단·문서화
- **음소거**: 🔊/🔇 토글 버튼(≥44px, 모바일 슬롯 규약(22장) 준수), localStorage 저장, 기본 ON
- **모바일 제약**: 오토플레이 정책상 첫 사용자 제스처(입장 버튼 클릭)에서 음성 엔진 프라이밍. iOS 무음 스위치·기기별 한국어 음성 유무는 문서화(없으면 조용히 무음 — 에러 금지)

## 31. 강의실 AI 조교 NPC (2026-07-20 발주자 요청)

- **기능**: 강의실 스크린 옆에 NPC("AI 조교")가 서 있고, 캐릭터가 다가가 말을 걸면 **화면 사이드에 1:1 개인 채팅 패널**이 열려 AI와 대화. 대화 내용은 본인에게만 보임(일반 채팅과 완전 분리, 다른 유저·서버 상태에 미노출)

- **NPC 렌더**: 마법사 모델 재사용(학자 실루엣, 고정 틴트) + Idle 애니 + "AI 조교" 닉네임표 — 스크린 남측 옆(x≈28.6, z≈-5.6), 강의실을 바라봄. 충돌 없음(장식), 네트워크·스키마 무변경
- **상호작용**: 반경 3m 근접 시 안내(데스크톱 `T 대화` 힌트 / 터치 `🤖 대화` 버튼 — 앉기 버튼과 같은 슬롯이나 좌석·NPC가 물리적으로 멀어 동시 표출 불가) → 패널 열림. 닫기 = X·Esc·6m 이상 이탈. 입력 중은 기존 uiCapture 규약(이동 키 정지)
- **대화 규약**: 클라이언트가 최근 12턴만 유지·전송(컨텍스트 상한), 메시지당 500자 제한, 서버 IP당 10회/분 레이트리밋, 20초 타임아웃. 페르소나: 최무호 월드 강의실의 친절한 한국어 조교(해요체, 2~4문장, 수업·월드 안내). 키 미설정/오류 시 정중한 한국어 안내 문구
- **낭독**: 조교 응답은 기존 TTS 큐로 낭독(거리 0, 🔊 음소거 토글 준수)
- **비용 고지**: Groq 무료 티어 한도 내 사용(초과 시 요청만 실패, 과금 없음 — 키 등급에 따름)
- **후속 1 — 자연스러운 음성 (발주자 요청)**: 조교 응답 낭독을 브라우저 Web Speech에서 **Edge TTS 신경망 여성 음성(ko-KR-SunHi)**으로 교체 — 서버 합성 프록시 `POST /api/npc-voice`(텍스트→MP3, msedge-tts, 무료)로 모든 기기에서 동일한 자연 음성. IP당 20회/분·600자 제한·15초 타임아웃·인사말 FIFO 캐시(50), 실패 시 Web Speech 폴백. 패널 닫으면 낭독 중단. 일반 채팅 낭독은 기존 Web Speech 유지
- **후속 2 — NPC 다인 확장 (발주자 요청)**: 방마다 조교 배치, 겉 명찰은 전부 "AI 조교", **이름은 대화로 물어봤을 때만 답함**(서버 페르소나 주입). 이름은 발주자 최종 확정(34장 후속): **강의실 = 아르티(Arty)**(오렌지, 스크린 옆) / **로비 = 노바(Nova)**(그린, 서벽 쪽에서 중앙 소파 세트를 바라봄) / **갤러리 = 루미(Lumi)**(블루, 전시실 동측에서 문 쪽을 바라봄) / **미로 = 큐리(Curi)**(보라, 골 챔버)
  - NPC별 독립 1:1 대화 이력(클라 세션 메모리), 요청에 npc id 포함(서버 검증), 배치는 검증 통행 경로·스폰 여유를 침범하지 않음(맵 불변식 테스트로 보장)
