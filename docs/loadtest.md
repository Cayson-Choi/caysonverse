# 100봇 부하 테스트 (Task 12)

- 실행일: 2026-07-14
- 결론: **통과 (로컬 기준)** — 아래 "판정" 참고. 로컬 수치는 참고용이며 최종 게이트는 Railway 배포 후 재측정한다.

## 1. 어떻게 실행하나

```bash
# 1) 프로덕션 번들 빌드 + 서버 기동 (tsx 오버헤드가 측정을 오염시키지 않도록 번들 실행)
npm run build
NODE_ENV=production node server/dist/index.cjs   # PORT 기본 2567, ADMIN_CODE는 설정하지 않음

# 2) (다른 터미널) 서버 리소스 측정 — 서버 PID는 작업관리자나
#    Get-NetTCPConnection -LocalPort 2567 -State Listen 으로 확인
node loadtest/measure.mjs --pid <서버PID> --duration 660 --interval 5 --out measure.csv

# 3) (다른 터미널) 100봇 실행 — 630초 (기본 4프로세스 샤딩, 아래 3절 참고)
npm run loadtest -- --numClients 100 --duration 630 --endpoint http://localhost:2567
```

- `npm run loadtest`는 `loadtest/run.mjs`(샤드 런처)로 인자를 그대로 넘긴다(pass-through).
  옵션: `--numClients`(기본 100), `--shards`(기본 4), `--duration`(초, 기본 600), `--endpoint`, `--statusDir`.
- 봇 스크립트를 소규모로 직접 돌릴 때는: `npm run bot -w loadtest -- --room world --numClients 10 --endpoint http://localhost:2567`
  (@colyseus/loadtest `cli(main)` 패턴 — `--room world`가 필수다.)
- 봇 행동: 10Hz 합법 이동(공유 `resolveCollision`으로 장애물 슬라이드, 매 틱 서버 권위 위치에 재고정),
  봇의 20%는 8~15초마다 채팅 1건, 30%는 5~10초마다 이모지 1건 — 전부 서버 rate limit 안쪽이다.

**100% = 코어(vCPU) 1개 포화** — 설계 게이트의 "1 vCPU의 50% 미만" 프레임과 동일 단위다.

## 3. 진단 여정 — 부하 생성기 쪽 함정 2가지 (서버는 내내 멀쩡했다)

@colyseus/loadtest는 한 프로세스 안에 numClients개의 연결을 만든다. 첫 시도들에서 봇이
집단 절단되거나 이동이 멎는 현상이 나왔는데, 진단 결과 **모두 서버가 아니라 부하 생성기
쪽** 문제였다 (문제 시점 서버 CPU ~2.5%):

**(1) @colyseus/loadtest TUI의 O(n²) 로그 누적 → 집단 절단.** 봇이 `room.onMessage`를
등록하지 않으면 SDK가 수신 브로드캐스트(채팅·이모지)마다 `console.warn`을 호출하는데,
loadtest의 `cli()`는 console.warn을 가로채 **계속 자라는 TUI 로그 문자열에 앞붙이기 +
전체 화면 재렌더**를 한다. 시간·클라이언트 수에 대해 O(n²)로 비대해져 이벤트 루프를
질식시키고, WS pong이 서버 ping 정책(3초 × 2회 ≈ 6초)을 놓쳐 **프로세스의 전체 봇이
한꺼번에 1006으로 강제 종료**된다 (100봇 단일 프로세스: 조인 완료 약 6초 만에 전멸;
25봇×4: 순단 248/재접속 216 폭풍; 10봇×10: 약 2분부터 서서히 붕괴). 수정: bot.ts가
와일드카드 핸들러 `room.onMessage("*", ...)`를 등록 — 실제 클라이언트도 어차피 구독한다.
수정 후에는 **단일 프로세스 100봇도 코어의 ~8%로 안정** 동작한다.

**(2) 직선 랜덤워크는 이 맵에서 랜덤워크가 아니다 → 이동 정지.** 목표가 칸막이벽 반대편에
있으면 봇이 벽을 따라 미끄러지다 z가 목표와 같아지는 순간 벽을 미는 상태로 **영원히
정지**한다(도착 판정 불가 → 새 목표도 안 뽑음). 수 분 안에 100봇 대부분이 x=±0.65(칸막이
면)에 얼어붙어 패치 트래픽이 클라이언트당 ~0.4KB/s로 붕괴한 것을 관찰자 프로브로 확인했다.
수정: bot.ts가 칸막이 반대편 목표는 **문(z∈±2) 경유 웨이포인트**로 라우팅하고, 30틱 동안
0.05m 미만 이동 시 목표를 다시 뽑는 안전망을 둔다.

`run.mjs`는 봇을 여러 OS 프로세스로 나누고(기본 4샤드 — 수정 후 필수는 아니고 안전 마진)
닉네임 오프셋(CV_BOT_OFFSET)을 부여한다 — 같은 머신에서 도는 로컬 실행이며 분산 부하
생성이 아니다. 실제 서비스의 브라우저 클라이언트 1명은 자기 연결 1개만 디코딩하므로
이런 생성기 함정과 무관하다.

## 4. 결과 (100봇, 630초 실행 — 부하 구간 615초)

- 실행 구성: 100봇 / 4샤드 / 630초, 서버는 프로덕션 번들(NODE_ENV=production), 같은 머신 loopback.
- 측정 원본: `.superpowers/sdd/evidence/task-12/measure.csv` (5초 간격 125샘플). 커밋에는 포함하지 않음.
- 아래 CPU/RSS는 부하 구간(15–630초, 116샘플) 기준. 전체 구간 요약(평균 4.3% / p95 11.5% / 최대 RSS 133.8MB)과 사실상 같다.

### 서버 리소스 vs 설계 게이트

| 지표 | 측정값 | 게이트 | 판정 |
|---|---|---|---|
| 평균 CPU | 4.6% | — | — |
| p95 CPU | 12.1% (최대 15.9%) | < 50% (1 vCPU) | **통과 (p95 기준 약 4배 여유)** |
| 최대 RSS | 133.8MB | < 400MB | **통과 (약 3배 여유)** |
| 클라이언트당 다운스트림 | ≈ 6.3KB/s | ≤ 30KB/s | **통과 (약 4.8배 여유)** |

### 봇 통계 (630초)

| 항목 | 값 |
|---|---|
| join 성공 / 시도 | 100 / 100 |
| join 실패 | 0 |
| 테스트 중 절단(disconnect) | 0 |
| 순단/재접속 | 0 / 0 |
| 방 오류(onError) | 0 |

다운스트림은 봇이 raw WebSocket 수신 바이트를 직접 합산한 값(전체 수신 386.3MB ÷ 100클라이언트
÷ 630초)이다. 전원이 계속 이동하는 최악 케이스 기준이며, 설계 문서의 예측(~24KB/s)보다 낮게
나온 것은 @colyseus/schema v4의 델타 인코딩이 이동 플레이어당 패치 ~10B 수준으로 설계 당시
가정보다 훨씬 조밀하기 때문이다(2클라이언트 프로브로 별도 확인).

## 5. 판정과 주의사항

- **판정: 통과 (로컬 기준).** 100봇 × 10분 이상, 절단·오류 0으로 세 게이트 모두 큰 여유로 통과.
- **로컬 ≠ Railway.** 이 수치는 Ryzen 7 6800H 데스크톱급 코어 기준이다. Railway 1 vCPU는
  이보다 느릴 수 있으므로 여유 배율을 그대로 믿으면 안 된다. 다만 측정된 마진(p95 CPU 기준
  약 4배)이 커서, 30분 게이트(사전 런칭 게이트)도 Railway에서 통과할 가능성이 높다.
- 30분 게이트는 이 태스크의 범위가 아니다(브리프: 로컬 10분이 이번 증거). 배포(Task 13) 후
  동일 스크립트로 30분 실측할 것.
- 서버와 봇이 같은 머신이라 네트워크 지연·패킷 손실이 0이다. 실환경에서는 재접속 로직이
  더 자주 개입할 수 있다.
- 봇 10Hz 틱은 setInterval 지연으로 실효 ~9.2Hz다. 이동 스텝은 틱 간격이 아니라 스텝
  길이 기준으로 합법이므로 서버 검증과는 무관하다.
- ADMIN_CODE 미설정 상태로 실행 — 관리자 경로(공지/강퇴)는 부하 시나리오에 포함되지 않는다
  (실사용에서도 관리자는 1명이라 부하 기여가 무시할 수준).
